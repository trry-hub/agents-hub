# Agents Hub 未来一年发展方向与当前不足

## 项目负责人判断

Agents Hub 不应该做成另一个普通聊天侧边栏，也不应该和 Copilot 在补全能力上正面竞争。它更适合成为 VS Code 里的 agent 工作台：统一接入 Codex、Claude Code、OpenCode、Gemini CLI 等工具。短期内先保证多个单 agent 的独立体验稳定；多 agent 并行、对比和编排作为后续阶段重新设计。

未来一年的核心目标是把 Agents Hub 从“多 CLI 入口”推进到“可视化 agent 控制台”和“自检自修复工作流”。用户打开它时，应该清楚知道四个问题：

- 当前可以交给哪些 agent 做事。
- 每个 agent 正在做什么、用了什么上下文、处于什么状态。
- 哪个结果可信，下一步应该执行、审查、测试还是回滚。
- 系统有没有主动检查自己的结果，并在失败时自动修复。

## 当前优势

- 已经具备多 provider 接入基础，可以在同一侧边栏内切换 Codex CLI、OpenCode、Claude Code 等工具。
- 已经具备多个 provider 的单 agent 切换基础，可以先把每个 provider 的官方式体验打磨稳定。
- 已开始支持任务状态、上下文摘要、模型、权限、运行方式等控制项。
- 输出清理已经能处理部分 CLI 噪声，例如 OpenCode banner、模型错误、Codex JSON 错误。
- 产品定位已经明确：不做补全，不复制 Copilot，而是做多 agent 编排和可视化监督。
- 已经具备任务状态和验证闭环的基础，可以进一步发展成自动自检、自修复的 agent 生命周期。

## 当前不足

### 1. 产品形态还不够稳定

当前界面仍有较多“边做边补”的痕迹。任务状态、agent 选择、模型选择、权限选择、上下文窗口等控件都在输入区附近竞争空间，用户容易不知道每一块的主次关系。

需要明确输入区的层级：

- 第一层：输入任务和发送。
- 第二层：选择参与 agent、模型、权限、上下文。
- 第三层：查看运行状态和历史任务。

任务状态不应该常驻抢占输入区。它应该只在运行中显现，结束后退到历史或详情里。

### 2. 多 agent 并行需要重新设计，当前先暂停

原先的并行更接近把同一个 prompt 发给多个工具，容易打乱输入区和任务状态，也会掩盖单 agent 流程本身的问题。当前阶段应该先移除同时发送和计划对比入口，保证 OpenCode、Codex、Claude Code 等 provider 都能作为独立 agent 稳定工作。

后续如果恢复多 agent 并行，必须先补齐任务建模：

- 谁负责规划。
- 谁负责实现。
- 谁负责 review。
- 谁负责运行测试和修复。
- 多个结果如何对比、采纳和合并。

如果没有任务分工、结果对比和后续验证，并行只是“同时开多个聊天窗口”，应该保持关闭。

### 3. 结果可信度没有形成闭环

Agent 输出之后，用户仍然需要自己判断结果是否可靠。当前缺少完整的验证链路：

- 识别改了哪些文件。
- 自动运行配置好的 build、test、lint。
- 把失败结果转成下一轮修复任务。
- 把 diff 交给另一个 agent 审查。
- 记录每次任务的输入、上下文、输出、验证结果。

没有验证闭环，Agents Hub 很难从“好用的入口”升级为“可靠的工作台”。

### 4. 缺少持续自检和自修复机制

从真实使用者视角看，理想的 agent 不应该“写完就停”。它应该像一个负责任的工程同伴：做完后自己检查，发现问题后自己修复，再次验证，最后把结果和剩余风险讲清楚。

当前还缺少一条明确的自我修复链路：

- 运行前确认目标、上下文、权限和工作区状态。
- 执行后检查输出是否为空、报错、超时或被 CLI 噪声污染。
- 修改代码后自动检查 diff、类型、lint、测试和构建。
- 失败时自动生成修复任务，而不是只把错误抛给用户。
- 多次修复仍失败时，停止自动循环并说明阻塞原因。
- 结束前给出“已验证 / 未验证 / 仍有风险”的明确结论。

这会成为 Agents Hub 区别于普通聊天插件的关键能力：它不是只回答，而是不断检查自己、修复自己，直到达到可交付状态或诚实说明无法继续。

### 5. 官方能力嵌入不足

用户明确偏好官方体验。当前只能桥接部分官方扩展命令，无法做到真正内嵌官方面板。这个限制要被产品化处理：

- 能用官方命令的地方优先调用官方命令。
- 不能内嵌的地方明确标注为 CLI 模式或本地模式。
- 不伪装成官方面板，但视觉和交互要尽量贴近官方工具的使用习惯。
- 对每个 provider 给出“官方能力可用度”说明。

### 6. Provider 适配层需要工程化

不同 CLI 的参数、模型、权限、输出格式、token 统计、错误格式都不同。当前已有 profile 和 output formatter，但还需要更清晰的 adapter contract。

建议把 provider 能力拆成稳定字段：

- 是否支持模型选择。
- 是否支持权限模式。
- 是否支持本地 token 估算。
- 是否支持后台会话 attach。
- 是否支持 diff、review、test、plan 等能力。
- 是否支持自动修复循环和最大重试次数。
- 是否支持官方 VS Code 扩展桥接。

这样推荐、UI 展示和任务编排才不会写成越来越多的特殊判断。

### 7. UX 还没有形成统一设计语言

近期多次调整说明一个问题：当前 UI 功能增长快，但设计系统还没定住。需要建立几条硬规则：

- 输入区永远保持干净，避免塞太多 chip。
- 运行状态用轻量状态行，不用重卡片。
- Provider 颜色只用于小圆点或极少量强调，不污染整块布局。
- 弹窗必须可关闭、不会被裁切、点击外部可收起。
- 控件优先像 VS Code 原生工具，而不是网页后台。

### 8. 安全和权限表达还不够清楚

Agent 会执行命令、修改文件、读取上下文。权限 UI 不能只展示“默认”“自动”等短词，还需要让用户理解风险：

- 只读、工作区写入、完全访问权限的边界。
- 哪些 agent 会调用 shell。
- 哪些任务会发送上下文到云端。
- 当前模型和 provider 是否在本地处理。

这些信息不一定常驻展示，但必须在用户做高风险选择时清楚可见。

### 9. 任务完成标准还不够产品化

当前“运行结束”容易被用户理解为“任务完成”，但两者不是一回事。真正的完成标准应该包括：

- 用户目标是否被满足。
- 是否修改了预期文件。
- 是否通过必要检查。
- 是否留下未处理错误。
- 是否需要人工确认高风险操作。

后续 UI 应该把任务状态从“运行状态”升级为“交付状态”，例如：运行中、等待验证、修复中、已验证、需人工处理。

## 未来一年发展方向

### 第一阶段：0-3 个月，打稳工作台基础

目标是让 Agents Hub 成为稳定、清爽、可信的多 agent 入口。

重点工作：

- 收敛输入区 UI，建立统一设计规范。
- 完善 provider adapter contract，把模型、权限、运行方式、token、官方桥接能力标准化。
- 优化 agent 选择器，让用户在发送时选择参与者，而不是通过 tab 切换理解工作流。
- 任务状态只展示运行中任务，完成后进入历史或详情。
- 建立 provider 健康检查：是否安装、版本是否过低、登录是否可用、模型是否支持。
- 完善错误归一化，让常见 CLI 错误变成可读、可行动的信息。
- 为每次任务增加结束前自检：输出是否为空、是否报错、是否需要用户下一步操作。

阶段验收：

- 新用户能在 3 分钟内配置并成功发送第一个任务。
- 运行中任务状态清楚，但不干扰输入。
- 至少 3 个 provider 的模型、权限、运行方式展示稳定。

### 第二阶段：3-6 个月，在单 agent 稳定后重新做多 agent 对比

目标是让“多 agent”不只是多入口，而是可比较、可选择、可复用的工作流。

重点工作：

- 做出计划对比视图：同一任务让多个 agent 给出方案，并排查看差异。
- 给每个方案展示 agent、模型、上下文、耗时、退出状态。
- 支持把某个方案提升为执行任务。
- 支持“合并方案”提示词，把多个 agent 的优点汇总成最终计划。
- 给推荐逻辑增加解释：为什么推荐这个 agent。
- 保存常用任务模板，例如“先规划，再实现，再 review”。
- 在对比视图中展示每个 agent 的自检结果：是否发现风险、是否建议继续执行、是否需要补充上下文。

阶段验收：

- 用户能一键让两个以上 agent 并行给出计划。
- 用户能从对比结果进入下一步执行。
- 推荐结果有可解释理由，而不是只显示“推荐”标签。

### 第三阶段：6-9 个月，建立验证闭环

目标是让 Agents Hub 从“发任务”升级为“交付任务”，并形成自检、自修复、再验证的闭环。

重点工作：

- 检测 agent 运行前后的文件变化。
- 自动生成任务结果摘要：改了什么、失败了什么、下一步是什么。
- 支持配置 build、test、lint 命令。
- 运行后自动触发验证，并把失败结果转成修复任务。
- 支持 review agent：把 diff 交给另一个 agent 找风险。
- 建立任务详情页，记录输入、上下文、输出、diff、验证结果。
- 引入自修复循环：验证失败后，系统自动携带错误日志和 diff 重新交给合适的 agent 修复。
- 设置循环边界：最大修复次数、权限边界、失败原因、人工接管点。
- 增加“完成前检查清单”：目标、diff、验证、风险、剩余问题。

阶段验收：

- Agent 修改代码后，用户能看到结构化 diff 摘要。
- 至少支持一种自动验证命令。
- 失败结果能一键交给 agent 修复。
- 常见验证失败可以自动进入一次修复并重新验证。
- 每个任务结束时都有明确的“已验证 / 未验证 / 需人工处理”结论。

### 第四阶段：9-12 个月，做成可扩展平台

目标是让 Agents Hub 支持团队和高级用户自定义自己的 agent 工作流。

重点工作：

- 支持自定义 provider adapter。
- 支持团队共享 workflow 模板。
- 支持 provider 能力评分和历史成功率。
- 增加本地任务历史搜索。
- 增加 opt-in 使用指标，帮助评估任务成功率、失败率、耗时和复用率。
- 基于历史记录优化自修复策略，例如哪些 agent 更擅长修测试、修类型、修构建。
- 完善文档、设置页、故障排查和首次引导。

阶段验收：

- 高级用户可以接入自定义 CLI agent。
- 用户可以保存并复用多 agent 工作流。
- 团队能根据历史数据判断哪个 agent 更适合某类任务。
- 团队可以看到自修复成功率和人工接管原因。

## 关键产品原则

- 官方优先：能调用官方能力就调用官方能力，不能伪装官方。
- 控制权优先：推荐可以强，但用户必须容易 override。
- 状态透明：每次任务都要知道谁在做、用什么模型、什么权限、什么上下文。
- 验证优先：没有 build、test、review 的 agent 输出不能默认可信。
- 自检优先：agent 完成后必须先检查自己，再把结论交给用户。
- 自修复有边界：允许自动修复常见错误，但必须限制次数、权限和风险范围。
- UI 克制：功能可以强，但界面必须像工具，不像控制面板堆料。
- 可扩展：provider 差异要进入 adapter，不要散落在 UI 判断里。

## 未来一年最重要的指标

- 第一次成功发送任务所需时间。
- 推荐 agent 被采纳的比例。
- 多 agent 对比任务的使用比例。
- Agent 运行后的验证通过率。
- 失败任务被自动修复的比例。
- 自修复循环的成功率和平均修复次数。
- 任务最终状态中“已验证”的占比。
- 人工接管原因的分布。
- 用户手动切换 provider 的次数是否下降。
- 每周重复使用的 workflow 数量。

## 最大风险

- 官方扩展能力不开放，导致“真内嵌”无法实现。
- 支持太多 provider，维护成本失控。
- UI 承载太多控制项，变得比官方工具更复杂。
- Agent 输出不可验证，用户信任无法建立。
- 自修复循环失控，反复修改代码却无法收敛。
- 权限和云端上下文边界表达不清，带来安全顾虑。

## 近期 30 天优先级

1. 固化输入区设计规范，避免每加一个能力就挤压 composer。
2. 把任务状态条改成运行时辅助，不作为常驻任务板。
3. 梳理 provider adapter 字段，减少 UI 特判。
4. 完成计划对比视图的最小版本。
5. 增强错误归一化和版本诊断，尤其是 Codex、OpenCode、Gemini CLI。
6. 写清楚官方桥接边界，让用户知道什么时候是官方体验，什么时候是 CLI 模式。
7. 定义任务结束前自检清单，先覆盖输出为空、退出码非零、模型不可用、上下文不足四类问题。
8. 设计自修复循环的最小版本：一次失败、一次修复、一次复验，超过边界就请求用户接管。
